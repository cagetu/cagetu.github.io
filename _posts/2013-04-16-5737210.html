Designing a High-Performance OpenGL ES Application

요약하면, 잘 디자인 된 OpenGL ES 어플리케이션은 다음과 같은 것이 필요하다.
  • OpenGL ES 파이프라인에 병렬화를 잘 활용한다.
  • 어플리케이션과 그래픽 하드웨어 사이의 데이터 흐름을 관리한다.


이 디자인의 중요한 성격을 요약하면,
  • 최대한 스태틱 리소스를 사용하라
  • innter rendering loop는 다이나믹 리소스의 수정과 rendering command들을 제출(submit)하는 것 사이를 번갈아 다룬다. 한 프레임의 시작이나 끝에서를 제외하고 다이나믹 리소스를 수정하는 것은 피하도록 하라
  • 어플리케이션에서 렌더링 결과를 중간의 렌더링 결과를 읽어 들이는 것을 피하라
Inner Loop
  • inner rendering loop에서 어플리케이션은 (프로세스내에 OpenGL ES Object를 생성하거나 수정하는) 렌더링 리소스 업데이트와 그런 리소스를 사용하는 drawing command를 submitting 을 번갈아 처리한다. 
  • Inner loop의 목적은 CPU와 GPU가 병렬적으로 작업하기 위해 작업량을 균형을 맞추는 것, 어플리케이션과 OpenGL ES가 동시에 같은 리소스에 접근하는 것을 막는 것이다.
  • iOS에서, OpenGL ES 오브젝트를 수정하는 것이 frame의 시작이나 끝에서 수행되지 않을 때, 비싼 처리가 될 수 있다.
  • inner loop에 대해 한 가지 중요한 목적은 OpenGL ES에서 application으로 데이터를 역으로 복사하는 것을 피하는 것이다. 
  • GPU에서 GPU로 결과를 복사하는 것은 매우 느리다. 카피된 데이터는 현재 프레임 렌더링 처리의 일부로 나중에 사용된다면, 어플리케이션은 모든 이전에 제출된(submitted) drawing command들이 완료될 때까지 block된다.
Flush
어플리케이션은 필요한 경우에만 command를 flushing라고 동기화의 사용을 제한해야한다. command를 flushing과 synchronizing의 과도한 사용은 어플리케이션이 렌더링이 끝날 때까지 하드웨어를 기다리는 지연(stall)을 일으킨다.

이 상황이 실행하는 동안 하드웨어로 OpenGL ES가 command buffer를 제출하도록 요구한다.
  • glFlush 함수는 command buffer를 graphics hardware에 보낸다. 이는 command가 하드웨어에 제출될 때까지 block하지만, command에 대해서 실행이 끝나는 것을 기다리지 않는다.
  • glFinish 함수는 그래픽 하드웨어에서 모든 이전의 submitted command들에 대해 실행이 끝나기를 기다린다.
  • OpenGL state(glGetError()처럼) 를 검색하는 함수는 submitted command가 완료되는 것을 기다린다.
  • command buffer가 가득 찼다.
Using glFlush Effcectively
  • 어플리케이션이 특별한 OpenGL ES object를 사용하여 rendering command를 제출했고, 그리고 아까운 미래에 오브젝트를 수정할 것이라면, pending drawing command를 가진 OpenGL ES object를 수정하려고, 어플리케이션이 그 drawing command가 완료될 때까지 멈춰있는 상태라면. 이런 상황에서는 glFlush호출은 하드웨어가 즉시 command 처리를 시작하는 것을 확신할 수 있다. command buffer를 flushing 한 후에, 어플리케이션은 제출된(submitted) command들을 가지고 병렬로 수행할 수 있는 작업을 수행해야만 한다.

Avoid Querying OpenGL ES State
  • glGetError()를 포함한 glGetxxx() 호출은 상태 값을 검색하기 전에 이전 command들을 실행하는 것을 요구한다. 이 동기화는 병렬로 처리될 수 있는 기회를 줄임으로 CPU와 같은 방식으로 그래픽 하드웨어가 실행되도록 강요한다. 이것을 피하기 위해, 질의에 필요한 상태를 복사하게 어플레이케이션에서 유지하고 직접 접근하는 것이 OpenGL ES 호출하는 것보다 더 나을 것이다.
  • glGetError()의 경우에는 어플리케이션을 디버깅하는 동안에만 오직 사용하도록 한다. 릴리즈에서 과도하게 glGetError를 호출하는 것인 성능을 줄일 수 있다.
