<객체 지향 소프트웨어 구성>
1. 소프트웨어 품질
소프트웨어 엔지니어링의 목표는 최종 사용자의 관점과 소프트웨어 개발자의 관점에서 모두 우수한 소프트웨어를 만들 수 있도록 원조하는 것이다.

*최종 사용자 중심의 조건(코드를 사용할 클라이언트 포함)
: 소프트웨어가 빠르고, 신뢰할 수 있으며 사용하기 쉬워야 한다.
*내부 개발자 중심의 조건
: 소프트웨어가 읽기 쉽고 모듈식이며 구조화되어 있다.

소프트웨어 구성 목표가 클라이언트의 요구사항을 구축하는 것이므로, 최종 사용자 중심의 품질이 더 중요하다. 객체 지향 설계는 내적 품질을 다루기 위한 것이지만, 최종 결과가 다음과 같은 품질을 충족시켜야 한다.

**정확성: 작업들을 정확히 수행할 수 있는 소프트웨어의 능력
**견고성: 비정상적인 상황에서도 작동할 수 있는 소프트웨어의 능력
**확장성: 사양의 변경에 맞춰 소프트웨어를 쉽게 변경할 수 있는 용이함
**재사용성 : 일부 또는 전체를 새로운 응용 프로그램에서 다시 사용할 수 있는 소프트웨어 능력
**호환성: 소프트웨어 제품을 다른 제품들과 쉽게 결합할 수 있는 용이함
**효율성: 프로세서, 메모리 & 기억장치와 같은 하드웨어의 자원들의 적합한 사용
**이식성: 다양한 하드웨어 및 소프트웨어 플랫폼으로 쉽게 전송할 수 있는 용이함
**검증: 소프트웨어의 오류 감지 및 검색을 위한 테스트 데이터 및 절차 준비의 용이함
**무결성: 액세스 또는 변경이 의도적인지 여부에 상관없이, 무단 액세스 및 변경으로부터 다양한 구성 요소를 보호할 수 있는 소프트웨어 시스템의 능력
**용이성: 프로그램 실행, 입력 데이터 준비, 출력 데이터 변환, 예외 상황으로부터의 복구를 비롯하여 소프트웨어 사용법을 쉽게 학습할 수 있는 용이함

2. 모듈성
모듈은 자율적이고 유기적이며 응집성을 지닌 구조적인 패키지이다. 다음 기준은 모듈식 소프트웨어 구성 방법에서 제공할 기능을 결정하는데 도움이 될 것이다.

*분해력: 설계 방법이 솔루션이 개별적으로 수행될 수 있는 몇 개의 하위 문제들로 하나의 문제를 분해할 수 있도록 도와준다. ( 예: 하향식 설계 )
*구성력: 설계 방법이 자유롭게 결합되어 새로운 시스템들을 구성할 수 있는 소프트웨어 요소
들의 생산을 지원한다. ( 예: 연산 라이브러리 )
*이해력: 설계 방법이 인간에 의해 개별적으로 파악될 수 있거나 몇 개의 다른 모듈들과 함께 파악될 수 있는 모듈들의 생산을 도와준다.
( 예: 명확하게 지정된 익스포트된 함수들이 들어 있어 연결 시 다른 라이브러리가 필요하지 않는 하나의 라이브러리 )
*연속성: 문제의 사양에서의 사소한 변경이 하나 또는 몇 개의 모듈의 변경을 가져올 경우. 변경이 시스템의 구조에 영향을 미치게 해서는 안된다.
*보호: 모듈 방법은 모듈에서 실행 시 비정상적인 조건의 영향이 해당 모듈(또는 몇 개의 모듈)로만 제한된 상태로 유지되는 구조를 만들어야 한다.
( 예: 소스에서 입력 및 출력의 유요화. 이 조건은 추상 데이터 형식의 전제 조건과 사후 조건이다 )

위의 5가지 기준은 모듈성을 유지하기 위해 수행해야 할 5가지 원칙을 이끌어낸다.

**언어학적 모듈 단위. 모듈은 사용되는 언어에서의 구문 단위에 해당해야 한다.
( 분해력, 구성력 및 보호 )
**약간의 인터페이스. 모듈은 가능한 몇 개의 다른 모듈과 통신할 수 있어야 한다.
( 연속성, 보호 )
**작은 인터페이스. 두 개 모듈이 통신해야 하는 경우 이들 모듈은 가능한 적은 정보를 교환해야 한다. 이를 weak coupling이라 한다.
( 연속성, 보호 )
**명시적 인터페이스. 두 개 모듈이 통신할 때마다, 모듈의 텍스트에서 이 원칙이 명확히 지켜져야 한다. 이를 direct coupling이라 한다.
( 분해력, 구성력, 연속성, 보호 )
**정보 은닉. 모듈에 대한 모든 정보는 공용으로 선언되지 않는 경우 공개되지 않아야 한다.
( 연속성, 반드시 보호될 필요는 없음 )

개방-폐쇠 원칙
*개방 모듈: 모듈은 항상 확장할 수 있어야 한다. 예를 들면, 데어터 구조에 필드를 추가하거나 구조체에서 작동하는 새로운 함수를 추가할 수 있다.
*폐쇠 모듈: 모듈을 다른 모듈에서 사용할 수 있다. 이 모듈에서는 안정성을 강조하며 모듈이 잘 정의되어 있고, 안정된 인터페이스를 가지고 있다고 전제한다. 예를 들면, 이러 모듈은 라이브러리로 컴파일될 것이다.

모듈의 공용 인터페이스는 변하지 않지만 내부 구현이 변경되는 경우, 모듈은 개방 및 폐쇠 모듈(수정 되었지만 종속 코드는 재컴파일되지 않아야 함)로 간주될 수 있다. 그러나 모듈에 대한 대부분의 변경 사항은 새로운 기능을 추가하는 것이다. 개방-폐쇠 모듈에서는 상속 개념을 고려해야 한다.

3. 재사용성
재사용성은 소프트웨어 엔지니어링에 있어 기본 문제이다.

다음은 재사용 기능 구성 요소를 만들기 위해 해결해야 할 모듈 구조의 몇 가지 문제들이다.

*다양한 형식
: 모듈은 다양한 형식의 구조에 적용될 수 있어야 한다. 이 경우, 템플릿 혹은 매개변수화된 데이터 형식이 도움이 될 수 있다.
*다양한 데이터 구조 및 알고리즘
: 알고리즘에서 수행되는 작동은 데이터의 기본 구조에 의존할 것이다. 다양한 기본 구조의 처리 시 모듈을 고려해야 한다. 이경우, 재정의가 도움이 될 수 있다.
*관련 루틴
: 모듈은 기본 데이터 구조 조작용 루틴을 액세스할 수 있어야 한다.
*표현 독립성
: 모듈에서는 사용자가 구현 방법 또는 사용되었던 기본 데이터 구조를 몰라도 작업을 지정할 수 있어야 한다. 이 경우, 재정의와 다형성이 도움이 될 수 있다.
*하위 그룹 안에서의 공통성
: 공통성, 그 안의 공통성을 찾아내야한다. 한 개 블록에서 변경이 필요한 경우 아마도 다른 유사 블록에서도 변경이 필요할 수 있으며 이 경우 유지보수 작업에 많은 시간이 소요되므로 유사 코드 블록의 반복을 피한다. 기본 데이터 구조를 드러내지 않은 추상적인 인터페이스를 구축한다.

4. 객체 지향
객체 지향 설계는 유지해야 할 함수 보다는 모든 시스템이나 하위 시스템에서 조작하는 객체에 기초하여 소프트웨어 구조를 만든다.

**객체를 찾는 방법
: 잘 구성된 소프트웨어 시스템은 특정 세계적 측면에서 작동 모델로 볼 수 있다. 소프트웨어 객체는 단순히 실세계 객체를 반영할 것이다.
**객체의 설명 방법
: 객체를 설명하기 위한 일반적인 방법은 추상적 데이터 형식을 사용하는 것이다. 추상적 데이터 형식에 대한 설명에는

형식(추상 매개변수가 됨)
함수(적용되는 연산)
전제 조건(연산이 적용되기 전에 충족되어야 함)
사후 조건(연산이 적용된 후에 충족되어야 함)
원리(함수 구성의 작동 방법)

이 포함된다.

또, 객체 지향 설계는 추상 데이터 형식 구현물의 구조화된 모음으로서 소프트웨어 시스템을 구성하는 것이다.

**객체 지향 모듈 구조
: 시스템이 데이터 구조에 기초하여 모듈화된다.
**데이터 추상화
: 객체가 추상 데이터 형식의 구현으로서 묘사되어야 한다.
**자동 메모리 관리
: 프로그래머의 간섭 없이 기본 언어 시스템에서 사용되지 않은 객체의 할당을 해제해야 한다.
**클래스
: 모든 비단순 형식은 모듈이며, 모든 고급 모듈은 형식이다. 클래스는 모듈 당 한개의 클래스를 전형으로 구현된다.
**상속
: 상속 클래스는 다른 클래스의 확장 또한 제한으로서 정의된다.
**다형성과 동적 바인딩
: 프로그램 엔티티는 두 개 이상의 클래스 객체를 참조할 수 있어야 하며 작업은 다양한 클래스에서 다양한 실정을 갖출 수 있어야 한다.
**다중 및 반복 상속
: 하나의 클래스를 두 개 이상의 클래스로, 동일한 클래스를 두 번 이상 선언할 수 있어야 ㅎ나다.

문제는 이 내용에서 언급한 모든 다양한 기능들을 언어에서 지원할 수 있는가 없는가 하는 점이다. C++이 순수한 객체 지향 언어는 아니지만, 위에서 언급한 예들을 상당히 잘 지원하는 편이다. 그러나 성늘이 주요시되는 경우를 처리할 때는 약간의 융통성을 허용한다.
