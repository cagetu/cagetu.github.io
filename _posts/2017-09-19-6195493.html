--- layout: post title: "[Vulkan 일지] 렌더스테이트 관리" date: 2017-09-19 01:35:40 +0900 --- [Vulkan일지] Pipeline State????에서 작성한 것처럼 일단 PipelineState를 기능단위로 구분해서, 각 기능에 대해서 함수를 만들어줬다. 만들고 보니, DirectX에서 했던 것처럼 Device->SetRenderState() / Device->SetVertexDeclaration() 등의 모습으로 변경되었다. 이제 조금 내가 아는 모습이 만들어졌다. 물론 멀티쓰레딩을 고려하면 다르게 처리해야 할 수도 있겠지만, 이 인터페이스를 유지하면서 갔으면 좋겠다.

이제는 이전 API에서 하던 것처럼 렌더스테이트의 변경이 최소화되지 않도록 처리를 해줘야 한다. Vulkan의 경우에는 사실 런타임에서 PipelineState를 생성하는 방식이 아니라, 사전에 런타임에 사용할 모든 PipelineState 그러니까, 렌더스테이트 조합에 대해서 미리 생성해놓고, 런타임에 들어가기를 원한다. 복잡하게 생각할 것이 아니라, Shader Compile에 대해서 생각해보면 된다. "셰이더 조합과 셰이더 폭발"이라는 주제는 이미 우리에게 익숙하지만 굉장히 까다로운 문제인데, Vulkan / DX12에서도 동일하게 나타난다고 보면 된다.

Vulkan에서는 PipelineCache라는 객체를 지원해주는데, ShaderCache와 동일하게, 생성된 파이프라인을 캐쉬로 만들고, 사전에 미리 캐쉬를 읽어서 만들어 놓으면 된다는 의미이다. 

나는 일단 Pipeline 생성 전략은 Shader 컴파일 전략도 동일하다.

1. 렌더링할 PipelineState에 대해서 Pipeline 리스트에서 검색
2. 없다면 새롭게 Pipeline 을 생성하고, Pipeline에 추가. (더럽게 느리다고 한다.)
3. 추가할 때, 파이프라인 캐쉬에도 추가
4. 다음 번에 실행할 때에는 PipelineCache에서 읽어서 미리 Pipeline을 생성해 놓음.
5. 필요할 때 검색해서 생성된 Pipeline을 이용

VkPipeline 객체를 여러 쓰레드에서 동시에 사용이 될 수 있기 때문에, 잘 관리할 수 있는 방법이 필요하다. 이건 Valve의 Dota2의 발표자료를 참고해서 사용하도록 하자. 일단 시작을 싱글이니까...

내가 렌더스테이트를 관리하는 방식은 noerror님과 같이 개발했던 엔진에서 부터 이어져와서 이 방식으로 계속 사용할 생각이다.
(https://www.slideshare.net/noerror/06-8209872)

얼마전에 오픈소스 엔진은 BlueShift 엔진을 봤더니, 비슷한 방식으로 처리하신 것을 보고 참고하기도 했다.
(정말 오래전부터 블로그를 통해서 너무나 많은 것을 배우고 있습니다. 꾸벅~)

이렇게 되면, 아직은 싱글스레드로만 처리되겠지만, 우리가 익숙한 모습으로 렌더스테이트를 관리할 수 있게 된다. 

// 서로 다른 비트만 체크
unsigned long rs_xor = m_RenderState ^ rs;
if (rs_xor == 0)
return;

if (rs_xor&CULL_MASK)
{
switch (rs&CULL_MASK)
{
case CULL_CCW:
m_Driver->SetCullMode(CULLMODE::_CCW);
break;
case CULL_CW:
m_Driver->SetCullMode(CULLMODE::_CW);
break;
case CULL_NONE:
m_Driver->SetCullMode(CULLMODE::_NONE);
break;
}
}

if (rs_xor&ZDISABLE)
{
bool bDepthTestEnable = !(rs&ZDISABLE);
m_Driver->SetDepthTestEnable(bDepthTestEnable);
}

if (rs_xor&ZWRITEDISABLE)
{
bool bDepthWriteEnable = !(rs&ZWRITEDISABLE);
m_Driver->SetDepthWriteEnable(bDepthWriteEnable);
}
