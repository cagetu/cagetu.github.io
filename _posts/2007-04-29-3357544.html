생각에는 게임 엔진에서, 지원하지 않고, 게임에서, 이렇게 처리하도록 해주면, 될거 같은데... 대신 업데이트나, 이런 부분을 세밀하게 외부에서 호출할 수만 있도록, 잘 정리해놓으면.. 해볼만 할 듯...

간단하게, 3가지 아키텍쳐 모델이 있음.
1. Syncronous function paralled model
간단하게 이야기 해서, 하나의 모듈을 하나의 코어에 맡겨버려서, 독립적으로 작업을 하도록 만드는 것.
문제점은 한쪽이 부하가 걸리면, 다른 것도 대기해야 하고, 할당된 녀석들만 일을 하기 때문에, 노는 녀석들이 생길 수 있다는 것등등..

2. Asynchronous function parallel model
n개의 쓰레드를 열어두고, 독립적인 Task를 쪼개서, 병렬적으로 큐처리를 해서, 넘겨준다..
A, B, C, D, E,
main <-----------Render--------------><-----------Render-------------->
sub1 <-----B-----><--D-->
sub2 <--D--><-----B----->
sub3 <-----G-----> <-----G----->

Render 시점에서는, Task가 작업이 끝났는지를 기다리지 않고, 가장 최근에 작업이 된 것들을 가지고 랜더링을 한다. 즉, 업데이트는 가장 최근 데이터를 Render할 곳에, 리스트업(백업)한 후에, 업데이트를 하도록 설계..
-동기화를 잘 맞춰야 한다는 것.. 한번 꼬이기 시작하면, 계속 연쇄적으로 밀릴 수 있다고 생각됨..
-Task를 얼마나 잘 배분하나에 대한 문제를 잘 해결해야 함.. 한쪽으로, 부하가 큰 작업들이 밀리면, 상대적으로 남는 녀석들이 발생할 수 있음..

3. Data parallel model
가능한 같은 작업들끼리 object 개념으로 객체화 해서, 넘겨주고, 처리 되는 데로 랜더링 하는 방식.
아직은 하드웨어가 지원하기에는 무리가 있다나?
