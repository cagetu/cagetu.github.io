--- layout: post title: "[까먹지말자]스마트포인터" date: 2005-03-02 21:07:37 +0900 --- 스마트 포인터란?
원시 포인터 자료형( C++ 언어가 기본으로 제공하는 포인터 자료형 정도.. )을 흉내내면서, 거기에 더불어 부가적인 기능을 가지고 있는 C++객체로서, 주로 operator->, operator* 연산자를 구현해 주는 방법을 통해 포인터로서의 역할을 수행하는 객체를 말한다.

<특징>
*소유권의 관리 작업 대행
: 소유권을 관리하는 방법은 꽤 다양하여, 주어진 문제의 유형에 딱 들어맞는 방법으로 관리를 해 줄 수 있다.
어떤 종류의 스마트 포인터는 자동적으로 소유권을 이전해 줄 수도 있다. 즉, 이런 종류의 스마트 포인터를 복사하게 되면, 원래의 스마트 포인터는 null을 가리키게 되고, 복사를 받은 스마트 포인터가 해당 객체를 가리키게 됨과 동시에 할당된 메모리에 대한 소유권 또한 가기게 되는 것이다.
소유권 관리 작업의 한가지 예로 완전 복사(Deep Copy)를 할 때, 사용되는데,
Template
class SmartPtr
{
public:
    SmartPtr( const SmartPtr& rSrc ) : m_pPoint( new T(*rSrc.pointee) );
    ...

};
하지만, SmartPtr자료형의 객체를 복사한다고 했을 때, Widget에서 파생된 ExWidget클래스의 인스턴스를 rSrc가 가리키고 있다면, ExWidget 객체의 Widget에 해당하는 부분만을 복사하게 되므로, DeepCopy를 이용하여, 복사를 해주어야 함을 고려해야 한다.
개인적으로는 별로 쓸모있는 기능은 아니라고 생각한다. -_-

*참조 카운트 기능
: 앞에서 알아본 참조카운팅를 이용하여, 객체의 생명주기를 관리한다.
생각해 봐야 할 것은, SmartPointer 안에 Count Pointer를 둘것이냐 하는 것인데, 만약 SmartPointer 내부에 Count Pointer를 두게 된다면,
Template
class SmartPtr
{
    T* m_pPoint
    int* pCount
    ...

};
스마트 포인터의 메모리 사용량을 2배로 부풀리게 될 것이고, 카운터 변수가 heap에 할당할 때, C++가 제공하는 기본 할당자는 작은 메모리를 다루는데 눈에 띄게 속도가 느리며, 공간적인 낭비도 무시하지 못할 수준으로 발생할 수 있다.

그래서, 효율적인 방법은 카운터 변수는 포인팅 받는 객체 자신이 담고 있는 것이다. 이러한 기법은 Intrusive Reference Counting 이라고 부른다. 이렇게 하면, 보통의 포인터 크기 외에 추가적인 오버헤드를 가지지 않게 할 수 있다.

또, 참고 연결 리스트라는 방법도 있는데, 참조 카운트 숫자가 0이 되는 것보다는 그 시점이 언제인지를 검출해 내는 것을 알 수 있도록 해준다.
아직 어디에 써야 하는지를 정확하게 파악하지도 써보지도 않았기에 걍 이건 넘어가련다.

이 밖에도 여러 가지 기능을 부가 할 수도 있지만, 역시 필요한 기능만 추가해서 제일 좋은 것 같다.

내 경우 일단 참조 카운트를 이용해서, 객체의 생명주기를 관리하는데 주력하는데 사용하는데
함수 내에서,
Void main()
{
    // 리소스를 얻어올 때 참조 카운트 1 증가.
    Txtr* pTex = g_TxtrMgr.GetSrc( “A.bmp” );
    // 텍스쳐로 사용이 완료되었다면,사용자가 참조카운트를 1감소시켜줘야 한다.
    pTex->Release();

    // 쓰고 카운트만 증가 시켜 놓고 해제를 않하면, 추후 매니져가 어떤 결과를 가져 올지는 예측이 불가능하다.
    Return;
}

일일이 저렇게 가져다 사용하고, 사용이 끝났으니 지운다는 것을 매번 하기도 짜증나고 까먹을 수 있어서 불편하다는 생각이 들었다.

하지만, 스마트 포인터를 이용한다면...
void main()
{
    // 리소스를 얻어올 때 참조 카운트 1 증가.
    TxtrPtr TxtrSmartPtr = g_TxtrMgr.GetSrc( “A.bmp” );
    // 실제 리소스와 똑같이 사용
    TxtrSmartPtr …
    Return;
    // TxtrSmartPtr은 로컬 변수이므로, 자동으로 소멸 -> 참조카운트를 1 감소시킨다.
}
와! 정말 편하다.

클래스 내부에서 사용을 할 때,
class Test
{
    TxtrSmartPtr m_TexPtr;
}

// 소멸자
Test::~Test()
{

    // delete m_pTexPtr; 처럼 포인터라고 개념을 가지고 있으면,
    // 이런 실수를 종종 할 수 있다. 나만 그런가?! -_-;;
    // 스마트 포인터는 포인터가 아닌 객체이므로,
    // delete 대신 Release로 사용일 끝났다는 것을 알려주면 안된다.
    m_TexPtr->Release();

    // 여기서는 그냥 자동으로 스마트 포인터의 소멸자가 호출되도록 해주어야 한다.

}

이 밖에 참조 연결 리스트라는 방법도 있는데, 이것의 개념은 참조 카운트가 0으로 떨어지는 시점이 언제인지를 검출해 내자는 것이다.

자세한 내용은 Modern C++에 잘 나와 있으니, 읽어 보도록 하면 될것.
결론적으로, 참조 연결 리스트는 오직 자유 저장 공간이 매우 부족할 경우에만 사용되어야 하고, 그렇지 않은 경우에는, 참조 카운팅 기법을 사용하는 편이 더 낫단다.

스마트 포인터에 대해 많은 것을 알지는 못하지만, 객체의 생명주기를 확실하게 관리하는 것 하나에만 사용한다해도 정말 괜찮은 녀석인것 같다.
