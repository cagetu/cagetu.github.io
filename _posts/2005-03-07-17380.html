디자인 패턴을 팩토리랑 싱글톤 외에 별로 쓸일이 없다고 생각했는데,
공부하면서 정리해 두자..

<의도>
다른 객체에 접근하기 위해 중간 대리 역할을 하는 객체를 둔다.
객체의 접근을 제어하는 이유는 실제로 그 객체를 사용하는 시점까지 생성과 초기화에 드는 비용 및 시간을 연기해 보자는 것이다.

예) 문서 편집기
문서를 읽어 내는 것은 내부 그래픽 객체가 있든 없든 매우 빠르게 진행되어야 한다. 또한 문서가 읽혀지는 그 시점에서 모든 내용을 다 읽어 올 필요는 없다. 이미지의 모든 내용이 한꺼번에 한 문서에서 다 보여질 필요는 없기 때문이다.
이에 대한 해결책으로 그래픽 프록시라는 또 다른 객체를 사용할 수 있다. 이 객체는 실제 이미지에 대한 대역을 담당한다. 프록시는 이미지처럼 동작하고, 필요할 때 이미지의 인스턴스를 만들어 낸다.
문서 편집기는 Graphic 추상 클래스에 정의된 인터페이스를 통해서 이미지에 접근한다. ImageProxy는 요청에 따라 생성될 이미지를 위한 클래스로 디스크상에 존재하는 이미지의 파일 이름을 참조자로서 관리한다. ImageProxy 생성자에 파일 이름이 파라미터로 전달된다.

ImageProxy는 이미지를 둘러쌀 박스의 크기 정보와 실제 이미지에 대한 참조자 정보를 갖고 있다. 이 참조자는 실제로 이미지를 인스턴스화하기 전까지는 의미가 없다. Draw() 오퍼레이션에 의해 실제로 그리기 전에 이 이미지를 인스턴스화 한다. 그러므로 Get() 오퍼레이션은 인스턴스화 되었을 때만 이 요청을 실제 이미지에 전달하고, 그전에는 프록시가 이미지에 대한 크기 정보를 직접 제공한다.

<활용성>
1. 원격지 프록시: 서로 다른 주소 공간에 존재하는 객체에 대한 지역적 표현으로 사용
2. 가상 프록시: 요청이 있을 때만 필요한 복잡한 객체를 생성한다.
3. 보호용 프록시: 원래 객체에 대한 실제 접근을 제어한다. 이는 객체별로 접근 제어 권한이 다를 때 유용하게 사용할 수 있다.
4. smart 참조: 실제 객체에 대한 참조자 수를 저장하고 있다가 더 이상의 참조가 없을 경우 해당 객체를 자동으로 없앤다.


[런타임 프록시 구조의 가능 객체 다이어그램]

- 실제로 참조할 대상에 대한 참조자를 관리한다. RealSubject와 Subject 인터페이스가 동일하면 프록시는 Subject에 대한 참조자를 갖는다.
- Subject와 동일한 인터페이스를 제공하여 실제 대상을 대체할 수 있어야 한다.
- 실제 대상에 대한 접근을 제어하고, 실제 대상의 생성과 삭제를 책임진다.
- Proxy의 종류에 따라서 다음을 수행한다.
*원격지 프록시는 요청 메시지와 argment를 인코딩하여 이를 다른 주소 공간에 있는 실제 대상에게 전달한다.
*가상의 프록시는 실제 대상에 대한 추가적 정보를 보유하고 있어 실제 접근을 지연할 수 있도록 해야 한다.
*보호용 프록시는 요총한 대상이 실제 요청을 할 수 있는 권한을 갖고 있는지를 확인한다.

**Subject: RealSubject와 Proxy에 공통적인 이터페이스를 정의하고 있어, Real-Subject가 요청되는 곳에 Proxy를 사용할 수 있게 한다.
**RealSubject: 프록시가 대표하는 실제 객체이다.

이 정도면, 프록시에 대한 기본 개념을 알 수 있을 듯 하다.
정리하면서 느낀 것인데, 스마트 포인터의 상위 개념인 듯하다. 스마트 포인터를 잘 이용해서, 쓰면 특별히 부담없이 사용이 가능할 듯..

안녕하세요. 매크로 없는 메비~랍니다.

일단.. 이것이 문제인데 쓰레드를 써서 로딩하는 것이 정말로 효과가 있는지를 확인하는게 중요하겠죠..? 잘 셋팅된 Win2k 의 경우 HDD 에 대한 Full Scratch 를 행할 경우 제 시스템에선(테스트 당시) CPU 점유율이 30% 정도 밖에 되지 않았습니다. 그렇지만 DMA가 켜지지 않은 Win9x 계열의 경우 간단히 100%까지 올라갔습니다. 이것은 OS 에서 HDD를 어떻게 접근하느냐에 따른 문제라는 것이겠죠.

뭐 제가 생각했던 것은 Proxy 패턴을 활용한 방법입니다. Design Pattern 에 나오는 Proxy 패턴은 Wrapper 에 해당하는 클래스로 실제 클래스와 동일한 인터페이스를 갖지만 그 인스턴스는 알맹이가 비어 있고 어떠한 순간 데이터가 준비되거나 요구할때 그 데이터를 공급해 준다는 것이죠. 가령.. 이런 형태로 콜을 하겠죠.

CBitmap* bitmap = g_resMgr.LoadBitmap("behindloading.bmp");

while(true) {
bitmap->DrawDecal(0, 0, 100, 100);
}

이렇게 했을 경우 g_resMgr 은 CBitmap 에서 상속받은 CBitmapProxy 를 리턴하고 다른 쓰레드를 통해서(또는 다른 비동기적인 방법을 통해서) 파일을 로딩합니다. 일단 리턴된 CBitmapProxy 는 리소스 매니저에서 로딩이 끝날때까지는 실제적인 처리를 하지 않고 있다가 로딩이 끝나야 데이터가 존재하듯이 처리하게 되죠. 흠.. 코드로 보여주는게 편하겠죠.

CBitmap
{
public:
virtual void DrawDecal(int x, int y, int w, int h){ /* 실제 그리는 루틴이 들어간다.*/ }
virtual int GetWidth(){ /* 크기를 리턴한다. */ }
virtual int GetHeight(){ /* 크기를 리턴한다. */ }
};

CBitmapProxy : public CBitmap
{
public:
virtual void DrawDecal(int x, int y, int w, int h){ if(bitmap) { m_bitmap->DrawDecal(x, y, w, h); } }
virtual int GetWidth(){ if(m_bitmap) return m_bitmap->GetWidth(); else return 0; }
virtual int GetHeight(){ if(m_bitmap) return m_bitmap->GetHeight(); else return 0; }

protected:
CBitmap* m_bitmap;
};

위의 코드를 보면 간단히 보이죠..? CBitmapProxy 는 간단히 상속받아 실제로 m_bitmap 이 설정될때까진 아무 일도 안하다가 m_bitmap 이 정해지면 그림을 그리게 되죠. 그런데 여기서 문제가 되는 것은 GetWidth() 와 같은 함수입니다. 위에서는 m_bitmap 이 없을 경우 0을 리턴하게 하였지만 실제로 저렇게 만들면 골치아플때가 많죠. 가령 처음 로딩되어서 딱 한번 화면에 뿌리고 그 부분을 갱신하지 않게 될 경우 저런 비트맵이 로딩되기 전에 저 메소드를 호출하면 골치아파지죠. 이럴때 방법은 리소스가 확실히 로딩되게 기다리거나 아니면 처음 로딩할때부터 Flag 같은 것을 주어서 확실히 로딩이 되도록 하는 방법이 있죠.

이러한 방식으로 프로그램을 짤 때는 가능한 위와 같이 리소스 내부적인 정보에 의존하지 않는 프로그래밍이 중요하게 됩니다. 가령 CBitmap 의 멤버변수를 public 으로 만들어 놓고 멋대로 접근할 수 있게 해 놓으면 지금 CBitmap 이 상속을 받은 놈인지 Proxy 인지도 모른체 접근하게 되니 위와 같은 방식은 쓸 수 없습니다.

또 한가지 중요한 점은.. 흠.. 이게 더 중요한데.. HDD를 대체 얼마나 긁어야 하는지 입니다. ^_^ 2~3프레임동안 로딩이 진행되어야 할 정도로 데이터가 비 효율적으로 사용된다면(몇메가..? 10k 100k 정도는 정말 깜.빡.하면 읽을 수 있습니다. 이것도 최적화를 어떻게 하느냐에 따라서지만) 데이터의 크기를 조절하고 게임 시작전에 그러한 데이터를 미리 로딩해 두는 것이 당연히 필요하겠죠.
