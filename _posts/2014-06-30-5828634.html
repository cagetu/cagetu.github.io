위의 참고자료를 보면 실제 구현된 내용을 볼 수 있다. 대략 소개되는 allocator는 이러하다.
  • Linear Allocator
  • Stack Allocator
  • Scope Stack Allocator
  • Pool Allocator
여기에 기초가 되는 메모리 정렬 (바이트 정렬) 기능이 포함된다. 메모리 정렬이 중요한 이유는 요즘 나오는 CPU 상당수가 메모리 정렬이 제대로 지켜진 데이터 블록만 읽고 쓸 수 있기 때문이다. 예를 들어 메모리 주소가 4바이트 정렬되어 있다면 메모리 컨트롤러가 읽어 오는 데 아무 문제가 없지만, 똑같이 23비트 정수를 불러올 때 메모리 주소가 0x6A321173이었다면 메모리 컨트롤러는 이 일을 하기 위해 0x6A321170 블록과 0x6A321174 블록 두 개의 4바이트 블록을 읽어 들여야 한다. 이런 후 두 32비트 정수의 일부분을 마스크하고 시프트해서 마지막으로 목적지 CPU 레지스터에 놀리적 OR 연산으로 써 넣는다. 어떤 프로세서는 이렇게 해주지도 않고, 메모리 정렬이 맞지 않는 데이터를 읽거나 쓰려고 하면 그냥 쓰레기 값을 가지고 오고, 강제 종료 시켜버린다. 

일반적으로 mmo 프로젝트에서는 Pool Allocator를 많이 사용하지만, Game Engine Architecture 책에서도 소개한 것처럼 Linear Allocator / Stack Allocator의 경우에는 한 프레임 루프 단위에서 사용되는 내용들에 사용하면 효과가 좋을 듯 하다. 예를 들면, Command Buffer 라던지, Render List 같은 것들... 

멀티 쓰레드로 처리를 한다면, Linear Allocator를 2개를 사용해서, 이중 버퍼 할당자를 만들어, i번째와 i+1번째 프레임에서 사용하는 버퍼를 다르게 한다. 그러면 작업 결과는 비활성 버퍼에 글어 있기 때문에, 비동기 처리를 할 때 이전 프레임 정보를 사용할 수 있는 유리한 점이 발생한다.

또, 다수의 리소스 로딩이나 스크립트 파일을 읽어들을 때, 메모리를 잘 사용해야 하는데, Linear Allocator 형태를 사용하는 것도 좋을 듯 하다. 물론 적절한 양을 미리 할당 잡아야 한다는 것이 좀 걸리기는 하지만... 음.. 이걸 보완하기 위해 Page 관리가 들어가면 성능상의 이슈가 발생하니 적절히 잘 처리해야 하겠지?!

C++에서 메모리를 관리한다는 것(단편화, 메모리 릭, 캐쉬, 속도 등)이 쉽지는 않지만, 이 정도 allocator만 적절히 사용해도 꽤 괜찮게 관리가 가능하지 않을까 생각이 된다.
